Qué es

Herramienta para renovar los videos promocionales de las categorías de cursos de Ciberaula. A partir del transcript del video actual, un LLM lo reescribe para que sea más vendedor, lo estructura en escenas, y el pipeline genera el video combinando avatares (con su voz real) en distintos escenarios, mezclando planos fijos (talking-head) y escenas con movimiento.

Salida: video horizontal 16:9, ~1 minuto, un video por categoría de curso.

Pensado como producto: todo lo específico del cliente (credenciales, personajes, zonas) vive en .env + config.json + zones.json. Cambiar de cliente = cambiar esos archivos, sin tocar el código.

Flujo de inicio a fin

Cada etapa guarda sus artefactos en output/<runId>/, así que se puede reanudar cualquier etapa sin repetir las anteriores (idempotente: salta lo ya hecho). La etapa de voz es opcional: por defecto la sintetiza HeyGen; solo se usa ElevenLabs propio si providers.voice=elevenlabs.

Arquitectura

Todo lo externo está detrás de interfaces + una fábrica (src/providers/factory.ts); el orquestador (src/core/pipeline.ts) no conoce proveedores concretos. Proveedores:

El audio final de cada video lo descarga/ensambla el pipeline; ffmpeg hace el mux y el corte.

Dos rutas de render (por escena)

Talking-head

Plano fijo del avatar hablando, con gestos (Avatar IV). Mejor lip-sync, voz consistente y más barato. Es la ruta por defecto.

Necesita una imagen base (Nano Banana: avatar en la zona). La voz la pone HeyGen vía heygenVoiceId.

Dinámico

Avatar en movimiento (camina, se acerca, se gira…). Se activa cuando la escena tiene el campo motion. Usa Veo + voice changer.

No usa imagen base; Veo parte de la foto de referencia del personaje. La voz real se aplica con ElevenLabs.

En un mismo video conviven ambas: el pipeline enruta cada escena según tenga o no motion.

Setup inicial

npm install
cp .env.example .env                  # poner las API keys
cp config.example.json config.json    # cliente, personajes, voces
# colocar imágenes de referencia en Avatars/ (p. ej. elena.jpeg)

Requisitos: Node ≥ 18 y ffmpeg/ffprobe en el PATH (para mux, corte y normalización).

Voces

Qué voz suena en cada escena lo decide providers.voice. Cada personaje tiene su voz en config.json (los voice_id concretos viven ahí, gitignored; aquí van por nombre).

Modo actual: providers.voice = "elevenlabs" (voz unificada)

Todas las escenas usan la misma voz de ElevenLabs del personaje → identidad consistente.

Tipo de escenaMotor de vozModeloCómo
Talking-head (HeyGen)ElevenLabs TTS (texto→voz)eleven_multilingual_v2se genera el audio y se le pasa a HeyGen, que solo anima + lip-sync.
Dinámica (Veo/fal)ElevenLabs STS / voice changereleven_multilingual_sts_v2convierte el audio genérico de Veo a la voz real (preserva timing → mantiene lip-sync).

Ambas usan la misma elevenLabs.voiceId del personaje, así que es la misma persona. Matiz: TTS y STS no suenan idénticos (el STS conserva la prosodia/acento del audio de Veo) → ver Registro; la solución de raíz es Seedance con audio_urls (lip-sync a la voz TTS, sin STS).

Modo alternativo: providers.voice = "heygen"

Las talking-head usan la voz nativa de HeyGen (heygenVoiceId) y las dinámicas siguen con ElevenLabs STS → resultan dos voces distintas en el mismo vídeo (inconsistencia). Por eso el default de producción es "elevenlabs".

Permisos de la API key de ElevenLabs: necesita text_to_speech (talking-head) y speech_to_speech (dinámicas). Faltando cualquiera, la etapa correspondiente da 401 missing_permissions.

Ajustes por personaje en elevenLabs.params (stability/similarity_boost/style/speaker_boost); actualmente 0.33 / 0.75 / 0 / true.

Parámetros · zones.json (librería de escenarios)

Catálogo compartido de 15 zonas (id → descripción) reutilizable en todos los videos del cliente (oficina, sala de vidrio, corredor con skyline, recepción, set de marca…). Claude referencia zonas por id; el pipeline resuelve la descripción. Mantener fija da consistencia de marca y variedad entre videos.

Texto de marca: la IA no rotula texto exacto de forma fiable ("CIBERAULA" sale deformado). Las zonas de marca se dejan con la pared limpia y el logo real se superpone como overlay en la Etapa 2.

Catálogo visual de zonas

Mantener el catálogo al día (automatizado): la galería de arriba se genera entre los marcadores <!-- ZONES:START/END --> desde zones.json (no editar a mano). Tras cambiar zonas: 
# regenera las 15 imágenes + sincroniza la galería de esta guía
node scripts/gen-zone-previews.mjs

# regenera solo unas zonas concretas (+ sincroniza)
node scripts/gen-zone-previews.mjs ZONE_9 ZONE_11

# solo sincroniza el texto/galería de la guía (no genera imágenes, no gasta)

Así la imagen y la descripción de cada zona quedan siempre coherentes.

El ProductionSpec (v2)

Es el guion de producción que genera Claude (o se edita a mano). Estructura plana:

{
  "schema_version": 2,
  "video": { "client", "course_category", "aspect_ratio", "language", ... },
  "characters": { "<id>": { id, name, gender, age_range, wardrobe } },   // solo creativo
  "zones": {},                                  // normalmente vacío (vienen de zones.json)
  "base_images": { "IMG_1": { character, zone, framing, used_in_scenes:[...] } },
  "scenes": [{
    "id": 1, "base_image": "IMG_1", "dialogue": "...",
    "tone", "emphasis_words": [...],
    "motion_prompt": "...",      // gesto talking-head
    "expressiveness": "medium",  // energía talking-head
    "motion": "...",             // PRESENTE = escena DINÁMICA (Veo); su texto describe el movimiento
    "estimated_seconds": 4,
    "overlays": [...], "persistent_overlays": [...]   // Etapa 2
  }],
  "brand_assets": {}
}
Campo de escenaNota
base_imageReferencia a la toma (de ahí se derivan personaje, zona y encuadre).
motionSi está presente, la escena es dinámica. Su texto va al prompt de Veo (estilo plano medio: "entra caminando, se centra, abre brazos"…). Sin motion = talking-head.
motion_prompt / expressivenessSolo para talking-head (HeyGen).
overlays / transition_* / camera_movementSe conservan para la Etapa 2 (composición); no afectan el render del avatar. Tiempos de overlay relativos al inicio de la escena.

Acepta también el formato legacy/v1 (anidado, scene_id, heygen_motion_prompt): se adapta solo a v2 con src/core/legacyAdapter.ts / el comando convert.

Receta de escenas dinámicas

Cuando una escena tiene motion, el motor dinámico hace:

1 · Veo 3.1reference-to-video: foto del personaje + prompt (movimiento + diálogo) → video con lip-sync (voz genérica)
2 · Voice changerElevenLabs STS revoz a la voz real, preservando el timing (el lip-sync sigue cuadrando)
3 · Mux + trimswap de audio y recorte automático de la cola muda (Veo da clips de 8s fijos)
Clave de consistencia (actualizado): el motor por defecto en fal es image-to-video partiendo de la imagen base de la escena (mismo look, generada con Nano Banana). Esto pronuncia bien las siglas (ChatGPT, etc.) — al contrario que reference-to-video, que las dice mal. La advertencia antigua ("image-to-video distorsiona la cara") aplicaba a arrancar desde un frame lejano o de baja calidad; con la imagen base nítida y bien encuadrada no ocurre. El motion admite coreografía rica, pero el frame inicial manda (ver Aprendizajes clave).

El nº de escenas dinámicas se fija con dynamic_ratio (≈mitad por defecto; tope opcional max_dynamic_scenes); el reparto garantiza apertura + cierre dinámicas, con movimiento variado por escena.

Motor alternativo (video.dynamic.provider): los 3 pasos de arriba son la ruta Veo + STS (gemini directo o fal). Otras opciones: HeyGen Shots (heygen-shot) — avatar con voz nativa, sin STS (ver Prueba HeyGen Shots); y Sync (sync, el motor por defecto actual) — Veo para el visual + TTS de ElevenLabs y lip-sync en Sync, sin STS (ver Lip-sync (Sync)).

Aprendizajes clave (escenas dinámicas cinemáticas)

Hallazgos consolidados produciendo el vídeo all-dynamic de Ciberaula. Aplican a cualquier curso.

1 · El frame inicial manda en image-to-video. El movimiento se anima desde la imagen base. Para un travelling real (recorrer una mesa, cámara que entra por una puerta, levantarse y sentarse) el frame de partida debe estar compuesto para ese movimiento: si quieres que camine a lo largo de la mesa hacia cámara, arráncalo en el lateral de la mesa (no al fondo del centro, o "empuja"/deforma la mesa); si quieres "se gira al notar la cámara", arráncalo trabajando de lado, no de frente. Encuádralo con el campo base_image.framing — el generador ahora respeta un framing propio y no fuerza "plano medio de frente". Movimiento mínimo desde un plano medio frontal = parece talking-head.
2 · revise NO regenera la imagen base si el fichero ya existe. Para cambiar framing/zona de una escena dinámica hay que borrar images/IMG_X.png a mano para forzar la regeneración; si no, anima el frame antiguo (síntoma: el cambio "no se aplica").
3 · Prompts de movimiento cinemáticos. Combinar cámara (dolly, arco, travelling lateral, grúa, push-in) + acción del sujeto + desambiguar primer plano vs fondo: el modelo puede asignar la acción a una persona del fondo. Decir explícito "the FOREGROUND person is the only one who moves; coworkers in the background stay seated and do NOT move". Para cierres/CTA en estudio, omitir la cláusula de "coworkers" (la zona estudio ya lo gestiona).
4 · Duración: los clips dinámicos (Veo/fal) son ~8s fijos. Frases largas se atropellan o cortan; los talking-head (HeyGen Avatar IV) sí admiten frases largas. Al convertir TH → dinámica, acortar el diálogo a ≈16–20 palabras por escena para que entre natural en ~8s.
5 · Truco de audio (visual bueno + pronunciación mala). Si una toma se ve bien pero pronuncia mal, se puede muxear el audio de una toma anterior sobre el vídeo nuevo; si duran casi igual el lip-sync aguanta: ffmpeg -i video.mp4 -i audio.mp4 -map 0:v -map 1:a -c:v copy -shortest out.mp4.
6 · Pronunciación de marcas/siglas → reescribir el token. HeyGen decía "cibarala"; con "Ciber Aula" (dos palabras) lo pronuncia bien. Caso conocido: ChatGPT sale bien en image-to-video y mal en reference-to-video.
7 · Dúo de avatares con HeyGen Cinematic Avatar (validado). Pasar avatar_id como array [look1, look2]; cada uno habla con su voz nativa (sin STS). En el prompt se asigna la línea de cada uno y se piden entradas por los laterales hacia el centro, en ambiente de oficina. En el spec se documenta con el campo cast: [{character, dialogue}] (el catálogo lo muestra como DÚO con la línea de cada uno).
8 · Trampa al copiar un run (cp -r). El manifest.json guarda videos[].filePath absolutos al run origen; assemble seguiría usando los clips viejos (síntoma: "sigue saliendo la escena antigua"). Tras copiar: reescribir esos filePaths al run destino y, si quieres que el catálogo lo tome como representante del curso, actualizar también createdAt.
9 · Glitch en el último frame de clips dinámicos. A veces el último frame es una imagen distinta. Recortar los últimos frames y remontar: ffmpeg -i in.mp4 -vf "select=lt(n\,N-5),setpts=N/24/TB" -af "atrim=end=DUR" ....
10 · Saldo. Cada re-tirada en fal (~$1.35) y en HeyGen Cinematic (~$5, reserva ~$10) cuesta dinero real; iterar movimiento/encuadre consume rápido. Mantener un colchón antes de lanzar lotes.

Prompts del sistema

Los prompts que usa el pipeline (estado actual). Las partes entre {llaves} se rellenan en runtime. Fuente: src/prompts/scenePlanner.ts y src/core/pipeline.ts.

1. Claude — Guion (system prompt)

Eres un director creativo y COPYWRITER experto en spots publicitarios cortos con avatares que hablan a cámara. Tu objetivo es CONVERSIÓN: tomas el mensaje o el transcript y lo haces más vendedor (gancho fuerte en los primeros segundos, beneficios concretos, prueba social, urgencia, CTA claro) SIN inventar datos ni promesas falsas. Generas un GUION DE PRODUCCIÓN en JSON estricto. Devuelves SOLO JSON válido, sin markdown, sin ```.

"""{current_transcript?}"""

Por qué tropieza el TTS con las vocales pegadas

El audio de las escenas talking-head lo sintetiza un TTS (texto → voz). El fallo no es de HeyGen: es un caso difícil para cualquier motor de síntesis.

  • Sinalefa / hiato de vocales iguales. Cuando una palabra acaba en vocal y la siguiente empieza por la misma vocal, al hablar se funden en un solo sonido alargado: "aprendes a automatizar" → "aprendes-aa-utomatizar". El motor duda entre fusionarlas, separarlas con una micro-pausa o pronunciar ambas; como sus datos son inconsistentes, sale un borrón a medio camino.
  • La preposición "a" es el peor caso. Es monosílaba, átona y sin consonante que la ancle; al chocar con otra "a" inicial se la traga la palabra siguiente y se pierde la frontera entre palabras. Pasa igual con "de el", "que entra", "su último", "mi idea".
  • Cómo falla por dentro. El motor hace texto → fonemas → alineación de duración → vocoder. Sin un cambio acústico entre las dos vocales (no hay consonante que marque el límite), el alineador no detecta frontera y el vocoder comprime/emborrona ese tramo, a veces con un artefacto audible.

Por eso no se arregla con ajustes de voz: la defensa es redactar evitando la colisión. La regla del planner ya instruye a Claude para que genere el guion sin vocales idénticas pegadas, sin anglicismos ("copy", "engagement") y sin siglas sueltas de dos letras. Un locutor humano haría la sinalefa natural sin pensar; el modelo aún no lo hace de forma fiable.

Paralelización (la palanca de velocidad)

El 80% del tiempo son las 8 dinámicas (~14 min) + el dúo (~5 min), y hoy se generan una tras otra. Cada escena es independiente (escribe su propio scene-NN.mp4, sin estado compartido), así que se pueden lanzar a la vez.

1 · Concurrencia por etapaSustituir el bucle for…await por un pool: lanzar N escenas a la vez en images/voice/videos en vez de en serie.
2 · Tope de concurrenciaUn semáforo (p. ej. 4-5 simultáneas) para respetar los límites de los proveedores (fal/HeyGen) y la reserva de crédito.
3 · Tolerancia a fallosallSettled: si una escena falla, las demás siguen; se re-tira solo la que falló.

Con un tope de ~4-5 simultáneas, las 8 dinámicas pasan de sumarse (~14 min) a 2 tandas (~3,5 min), y el dúo corre en paralelo con ellas (~5 min). Las imágenes también se paralelizan (~3,4 min → ~1 min). El total bajaría de ~25 min a ≈ 6-7 min (≈ la cadena más lenta), sin perder calidad — la latencia individual la pone el servidor (HeyGen/Veo), pero se solapa.

A vigilar al paralelizar: límites de jobs simultáneos y rate-limits de cada API; la reserva de crédito de HeyGen (~$10+/job) exige más colchón de saldo si lanzas varios; logs que se entremezclan (prefijar por escena); y manejar el fallo de una escena sin tumbar el lote.
Cómo se implementaría: un helper de concurrencia (pMap con límite, sin dependencias) que envuelva los bucles de images()/voice()/videos() en src/core/pipeline.ts, con un parámetro concurrency en config.json (default 1 = comportamiento actual). Avanzado: pipelinear etapas (empezar el vídeo de una escena en cuanto su imagen base está lista, sin esperar a que terminen todas las imágenes).

El problema de la velocidad (pace del diálogo)

La clienta pidió un ritmo más rápido. Su referencia es el vídeo original puesto a 1.5× en su reproductor (VLC), que le suena bien. El reto: replicar ESE sonido en un archivo ya horneado.

Hallazgo central: acelerar en post (offline) NO suena igual que la reproducción en tiempo real de un player. El vídeo se generó una sola vez; la voz de las escenas dinámicas viene de Veo + voice-changer y no tiene un control de "velocidad de habla" en origen, así que la única palanca sobre el material existente es el time-stretch del audio. El resultado depende por completo de qué algoritmo se use.

Equivalencia velocidad ⇄ player: "+X%" = factor 1+X/100 = lo mismo que poner (1+X/100)× en un player. Es decir 1.5× = +50%, 1.4× = +40%, 1.3× = +30%.

Si atempo no llega a igualar al VLC de la clienta, la salida pragmática es entregar el original a velocidad normal y que ella lo vea a 1.5 en su VLC (el sonido exacto que validó), dejando el bake solo para la versión publicada. Hornear la velocidad en el pipeline está en el backlog (bloqueado hasta su OK).
